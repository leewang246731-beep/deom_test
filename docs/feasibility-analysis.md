# 可行性分析

---

## 一、架构变更评估

| 变更 | 复杂度 | 说明 |
|------|:---:|------|
| 删除 consumer 角色 | 低 | 移除注册/登录的消费者路径，所有用户改走商户员工认证 |
| 多租户数据隔离 | 中 | merchant_id 植入所有表和查询，ChromaDB Collection 隔离 |
| Platform Connector 抽象层 | 中 | 策略模式，Mock 一期完整实现，真实平台二期 |
| 客服工作台重构 | 中 | 原购物车/订单页面废弃，改为会话列表+聊天窗口+AI面板 |
| AI 从推荐→话术 | 低 | 原推荐引擎的向量检索能力完全复用，只是检索对象从"商品推荐"变为"话术检索" |
| Mock 演示数据 | 低 | Faker 库生成中文数据，种子脚本一键运行 |

---

## 二、技术复用

| 原系统能力 | 新系统用途 |
|-----------|-----------|
| ChromaDB 向量检索 | 商品语义搜索 + 话术检索 |
| BGE-M3 Embedding | 商品 + 历史话术向量化 |
| Redis 缓存/Session | 外部 Token 存储 + 分布式锁 |
| FastAPI + JWT | 不变，改动仅角色校验 |
| LangGraph/RAG | 简化为单品 LLM Pipeline，去掉多 Agent 编排 |
| Element Plus | Table/Form/Dialog 适用管理后台 |

---

## 三、风险评估

| 风险 | 缓解 |
|------|------|
| 多租户数据泄漏 | 所有 API 强制 `merchant_id` 过滤，单元测试覆盖 |
| Mock 数据不够真实 | Faker zh_CN + 真实商品品类名称模板 |
| AI 话术响应超过 2s | 向量检索 <500ms，LLM 流式返回 |
| 平台 API 变更（二期） | Connector 抽象层隔离变化 |

---

## 四、成本估算

| 阶段 | 人周 |
|------|:---:|
| 数据模型重构 + 多租户 | 3 |
| Platform Connector (Mock) + 种子脚本 | 2 |
| API 重构 | 3 |
| 前端重构 | 4 |
| AI 话术 Pipeline | 2 |
| 联调测试 | 2 |
| **总计** | **16 ≈ 4 周** |

运行成本不变：~¥1100/月（与重构前相同技术栈）

---

## 五、结论

方案可行。核心架构从"自营 B2C"切换到"多平台 SaaS"，技术栈零变化。一期 Mock 模式完整可演示，后续逐步接入真实平台。
